208   # Superclass for ActionController functional tests. Functional tests allow you to
 
209   # test a single controller action per test method. This should not be confused with
 
210   # integration tests (see ActionController::IntegrationTest), which are more like
 
211   # "stories" that can involve multiple controllers and multiple actions (i.e. multiple
 
212   # different HTTP requests).
 
213   #
 
214   # == Basic example
 
215   #
 
216   # Functional tests are written as follows:
 
217   # 1. First, one uses the +get+, +post+, +put+, +delete+ or +head+ method to simulate
 
218   #    an HTTP request.
 
219   # 2. Then, one asserts whether the current state is as expected. "State" can be anything:
 
220   #    the controller's HTTP response, the database contents, etc.
 
221   #
 
222   # For example:
 
223   #
 
224   #   class BooksControllerTest < ActionController::TestCase
 
225   #     def test_create
 
226   #       # Simulate a POST response with the given HTTP parameters.
 
227   #       post(:create, :book => { :title => "Love Hina" })
 
228   #
 
229   #       # Assert that the controller tried to redirect us to
 
230   #       # the created book's URI.
 
231   #       assert_response :found
 
232   #
 
233   #       # Assert that the controller really put the book in the database.
 
234   #       assert_not_nil Book.find_by_title("Love Hina")
 
235   #     end
 
236   #   end
 
237   #
 
238   # == Special instance variables
 
239   #
 
240   # ActionController::TestCase will also automatically provide the following instance
 
241   # variables for use in the tests:
 
242   #
 
243   # <b>@controller</b>::
 
244   #      The controller instance that will be tested.
 
245   # <b>@request</b>::
 
246   #      An ActionController::TestRequest, representing the current HTTP
 
247   #      request. You can modify this object before sending the HTTP request. For example,
 
248   #      you might want to set some session properties before sending a GET request.
 
249   # <b>@response</b>::
 
250   #      An ActionController::TestResponse object, representing the response
 
251   #      of the last HTTP response. In the above example, <tt>@response</tt> becomes valid
 
252   #      after calling +post+. If the various assert methods are not sufficient, then you
 
253   #      may use this object to inspect the HTTP response in detail.
 
254   #
 
255   # (Earlier versions of Rails required each functional test to subclass
 
256   # Test::Unit::TestCase and define @controller, @request, @response in +setup+.)

268   # == Testing controller internals
 
269   #
 
270   # In addition to these specific assertions, you also have easy access to various collections that the regular test/unit assertions
 
271   # can be used against. These collections are:
 
272   #
 
273   # * assigns: Instance variables assigned in the action that are available for the view.
 
274   # * session: Objects being saved in the session.
 
275   # * flash: The flash objects currently in the session.
 
276   # * cookies: Cookies being sent to the user on this request.
 
277   #
 
278   # These collections can be used just like any other hash:
 
279   #
 
280   #   assert_not_nil assigns(:person) # makes sure that a @person instance variable was set
 
281   #   assert_equal "Dave", cookies[:name] # makes sure that a cookie called :name was set as "Dave"
 
282   #   assert flash.empty? # makes sure that there's nothing in the flash
 
283   #
 
284   # For historic reasons, the assigns hash uses string-based keys. So assigns[:person] won't work, but assigns["person"] will. To
 
285   # appease our yearning for symbols, though, an alternative accessor has been devised using a method call instead of index referencing.
 
286   # So assigns(:person) will work just like assigns["person"], but again, assigns[:person] will not work.
 
287   #
 
288   # On top of the collections, you have the complete url that a given action redirected to available in redirect_to_url.
 
289   #
 
290   # For redirects within the same controller, you can even call follow_redirect and the redirect will be followed, triggering another
 
291   # action call which can then be asserted against.
 
292   #
 
293   # == Manipulating the request collections
 
294   #
 
295   # The collections described above link to the response, so you can test if what the actions were expected to do happened. But
 
296   # sometimes you also want to manipulate these collections in the incoming request. This is really only relevant for sessions
